### 第 2 页：项目背景与核心问题

**标题**：信息过载时代，如何让 AI 真正"理解"你的文档？

**左侧——两大痛点**：
```
📚 信息过载                          🔍 检索低效
• 学术论文、教材、文档数量爆炸        • 传统搜索只能关键词匹配
• 难以高效筛选和消化                   • 无法理解语义和上下文
• 知识碎片化，难以系统掌握            • 缺乏跨文档的关联能力
```

**右侧——核心命题**：
> RAG（检索增强生成）技术让 AI 能够"读懂"你的文档——先从知识库中精准检索相关内容，再基于检索结果生成准确回答。

### 第 7 页：混合检索深度解析

**标题**：BM25 + 语义检索 + RRF 融合 —— 稀疏与稠密的互补优势

**左侧——两种检索方式对比**：

| 特性 | BM25 关键词检索 | 语义向量检索 |
|:---:|:---:|:---:|
| 原理 | 词频统计 + 逆文档频率 | Embedding 向量余弦相似度 |
| 优势 | 精确匹配、专有名词、术语 | 语义理解、同义词、概念查询 |
| 劣势 | 无法理解同义词和上下文 | 对精确匹配不够敏感 |
| 实现 | jieba 分词 + rank-bm25 | ChromaDB + HNSW 索引 |

**右侧——RRF 融合算法**：

```
RRF_score(d) = Σ wᵢ / (k + rankᵢ(d))

其中：
• k = 60（平滑参数）
• w_semantic = 0.7（语义权重）
• w_bm25 = 0.3（关键词权重）
• rankᵢ(d) = 文档 d 在第 i 个排序中的排名

优势：
✓ 对分数绝对值不敏感，只关注相对排名
✓ 天然规避不同检索器分数尺度不一致
✓ 双路命中的文档获得加权加成
```

**底部——Cross-Encoder 重排序**：
> RRF 融合后的 top-20 候选送入 BGE-Reranker-v2-m3 精排，输出最终的 top-k 结果

### 第 8 页：关键技术实现 —— 降级设计 + 查询重写

**标题**：优雅降级优于完美运行 —— 每个组件都有 Plan B

**上半部分：三层降级策略**：

| 降级层级 | 触发条件 | 降级行为 | 用户体验影响 |
|:---:|:---|:---|:---:|
| L1 | BM25 索引未就绪 | 混合检索 → 纯语义检索 | 精度略降，核心功能不受影响 |
| L2 | Cross-Encoder 加载失败 | 重排序 → 返回 RRF 原始排序 | 精度降低但可用 |
| L3 | 查询重写 LLM 调用失败 | 回退 → 使用原始用户消息 | 多轮对话指代可能不准确 |

**下半部分：查询重写效果对比**：

```
❌ 无查询重写:
  用户第1轮: "请介绍柏拉图的洞穴喻言"
  AI: "洞穴喻言出自《理想国》第七卷..."
  用户第2轮: "它的核心论点是什么？"
  检索系统: 搜索 "它的核心论点" → 无相关结果 ❌

✅ 启用查询重写:
  用户第2轮: "它的核心论点是什么？"
  查询重写: → "柏拉图的洞穴喻言的核心论点是什么？"
  检索系统: 搜索完整的独立查询 → 精准命中 ✅
```

**底部金句**：
> "在 AI 应用中，每个外部依赖都可能不可靠。好的系统不是没有故障，而是每个组件都有降级路径。"

### 第 10 页：双模式对话设计

**标题**：为什么需要"苏格拉底式"？—— 引导式提问促进深度学习

**左侧——两种对话模式对比**：

| 模式 | 适用场景 | 特点 |
|:---:|:---:|:---|
| 直接问答 | 快速查询、事实性问题 | 基于知识库直接给出准确答案 |
| 苏格拉底式 | 深度学习、理解性问题 | 通过追问引导思考，促进知识内化 |

**右侧——为什么要做苏格拉底式？**

```
传统问答模式的问题：
┌────────────────────────────────────────────┐
│ 用户: "什么是机器学习？"                      │
│ AI: "机器学习是人工智能的一个分支..."           │
│ 用户: "哦，知道了" [被动接收，缺乏思考]         │
└────────────────────────────────────────────┘

苏格拉底式引导：
┌────────────────────────────────────────────┐
│ 用户: "什么是机器学习？"                      │
│ AI: "你觉得人类是如何学会识别猫的？"           │
│ 用户: "通过看很多猫的例子？"                   │
│ AI: "没错！那如果让计算机也通过看例子来学习呢？" │
│ [引导用户自己推导出概念，促进深度理解]          │
└────────────────────────────────────────────┘
```

**底部——设计原则**：
> 苏格拉底式模式不是"不给答案"，而是**先引导思考再给答案**。通过精心设计的提示词框架（身份层+规则层+方法论层），让 AI 在适当场景选择引导式提问。

│  1. 不要内联引用（由系统侧面板展示）        │
│  2. 格式友好（善用 Markdown）              │
│  3. 语言自适应（与提问者相同语言）          │
├──────────────────────────────────────────┤
│  🎯 方法论层                              │
│  苏格拉底模式: 追问引导、从具体到抽象       │
│  直接问答模式: 知识库优先、简洁明了         │
├──────────────────────────────────────────┤
│  📚 知识融合层                             │
│  • 知识库优先（用户提供的权威资料）          │
│  • 训练知识补充（过时时标注 ⚠️ 修正）       │
│  • 网络搜索规则动态注入                    │
└──────────────────────────────────────────┘
```

**设计原则**：
- **行为约束优先**：明确"不能做什么"比"应该做什么"更有效
- **动态注入**：网络搜索规则按需追加，不污染基础提示词
- **双模式隔离**：不同场景使用不同提示词，避免风格混淆

### 第 13 页：AI Agent 协作开发策略（二）

**标题**：AI 生成的代码不总是对的 —— 迭代修复闭环 + 代码审查

┌─────────────────────────────────────────────┐
│  🔄 AI Agent 协作的 Bug 修复闭环             │
│                                             │
│  ① 精确描述现象                              │
│  "上传 PDF 后文档状态一直是 processing"       │
│          ↓                                  │
│  ② Agent 搜索根因                            │
│  "搜索所有设置 document.status 的地方"        │
│          ↓                                  │
│  ③ Agent 提出修复                            │

│  "上传同一份 PDF，确认状态变为 ready"          │
│          ↓                                  │
│  ⑤ 回归检查                                  │
│  "检查这次修改是否影响文档归档功能"            │
└─────────────────────────────────────────────┘
```

**右侧：两种质量保障策略**

| 策略 | 操作方式 | 关键 Prompt |
|:---|:---|:---|
| 🔍 代码审查 | 新对话中让 Agent 以 Reviewer 视角检查 diff | "请审查当前分支的改动，重点关注安全性、错误处理" |
| 📋 上下文管理 | 阶段性总结 → 新建对话 → 附上总结 | "请总结当前进度，列出已完成功能和待完成功能" |

**底部——关键洞察**：
> 当 Agent 生成的代码有 Bug 时，**把错误现象反馈给 Agent 让它自己修复**——而不是手动修完再告诉它。

## 附一：PPT 设计建议

### 配色方案
- **主题色**：深邃蓝渐变（`#0f172a → #1e1b4b`），呼应项目 Cosmos 主题
- **强调色**：蓝色系（`#6366f1` / `#818cf8`），用于标题和关键数据
- **代码块**：深色背景 + 高亮语法
- **成功/完成**：绿色（`#10b981`），用于完成度标记和 checkmark

### 排版原则
- **标题即结论**：每页标题直接传达核心观点，听众只看标题就能理解
- **图表优先**：工作流用流程图、对比用表格、架构用框图，减少密集文字
- **关键数据突出**：40 个端点、6 个页面、30+ 提交等数字用大字体+颜色强调
- **代码精简**：只展示关键逻辑片段（5-10 行），不放完整源码
- **截图有标注**：每张截图需用红框/箭头标注要展示的关键区域

### 动画建议
- 流程图使用**逐级出现**（Build-in）动画，引导听众跟随讲解节奏
- 数据指标使用数字滚动动画
- 页面切换使用淡入淡出，避免花哨的 3D 翻转效果
- 截图可使用平滑放大过渡
