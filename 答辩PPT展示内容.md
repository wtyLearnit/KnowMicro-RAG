### 第 2 页：项目背景与核心问题

**标题**：信息过载时代，学习者需要的不只是"答案"，而是"思考的能力"

**左侧——两大痛点**：
```
📚 信息过载                          📖 被动接收
• 学术论文、教材、文档数量爆炸        • 搜索引擎 = 答案灌输
• 难以高效筛选和消化                   • 缺少引导学生主动思考
• 传统搜索只能"找"不能"懂"            • 知识从被动接收到主动发现的鸿沟
```

**右侧——核心命题**：
> 能否让 AI 扮演苏格拉底的角色——不直接给出答案，而是通过追问引导学生从具体现象出发，自主发现底层原理？

**底部——开发方式特色**：
> 💡 本项目全程使用 **Claude Code（AI Agent 工具）** 进行 Vibe Coding 协作开发

### 第 5 页：核心工作流一 —— 文档索引流水线

**流程图（纵向）**：

```
📄 用户上传文件 (PDF/DOCX/TXT/MD/PPTX, ≤50MB)
          │
          ▼
┌──────────────────────┐
│ 阶段① 文档解析        │  PyPDF2 / python-docx / python-pptx / UTF-8
│ 按扩展名选解析器       │  PPTX: 逐页提取标题+文本，保留幻灯片结构
└──────────┬───────────┘
           ▼
┌──────────────────────┐
│ 阶段② 智能分块        │  🔑 技术亮点：自适应策略选择 + 安全兜底
│ ├─ Markdown 感知 (md) │  按 #/##/### 标题层级，保持文档结构
│ ├─ 语义边界 (txt/pdf) │  按中英文句子边界，保持语义连贯
│ └─ 递归分割 (回退)    │  段落→换行→句号→短语→空格 逐级降级
│ 参数：800 字符/块     │  150 字符重叠 | 过滤 ≤10 字符碎片
│ 安全：>6000 字符强制  │  二次分割，防止超长块
└──────────┬───────────┘
           ▼
┌──────────────────────┐
│ 阶段③ 双路索引        │  🔑 技术亮点：冗余检索保障 + 健壮性设计
│ ┌──────────────────┐ │
│ │Embedding 向量化   │ │  批量 API 调用 (10条/批)
│ │→ ChromaDB 存储    │ │  指数退避重试 (0.5s→1s→2s, 最多3次)
│ │余弦距离 + HNSW    │ │  自动过滤空文本防 API 报错

└──────────┬───────────┘

```

### 第 6 页：核心工作流二 —— 混合检索与 RAG 对话

**标题**：七步 RAG 流水线 —— 查询重写→双路检索→RRF融合→重排序→上下文构建→流式生成

**完整流程图**：

```
👤 用户输入问题
        │
        ▼
┌─────────────────────────────────────────────┐
│ Step 0: 消息持久化 → SQLite                  │
└─────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────┐
│ Step 1: 查询重写 (LLM)              🔑 亮点  │
│ "它的主要论点是什么？"                          │
│   → "《理想国》的主要论点是什么？"               │
│ temperature=0.1, max_tokens=200              │
│ ⚠️ 失败自动回退原始消息（非关键路径，不阻塞对话）  │
└────────────┬────────────────────────────────┘
             │
    ┌────────┴────────┐
    ▼                 ▼
┌──────────┐   ┌──────────────┐
│Step 2a:  │   │Step 2b:      │
│语义检索   │   │BM25 关键词检索│
│ChromaDB  │   │jieba 分词     │
│余弦相似度 │   │BM25Okapi 评分 │
│top-20    │   │top-20         │
└────┬─────┘   └──────┬───────┘
     └──────┬─────────┘
            ▼
┌─────────────────────────────────────────────┐
│ Step 3: RRF 融合 (Reciprocal Rank Fusion)    │
│ RRF(d) = Σ wᵢ/(k + rankᵢ(d)), k=60         │
│ semantic=0.7  BM25=0.3                      │
│ BM25 分数归一化到 [0,1] · 保留语义相似度展示    │
└────────────┬────────────────────────────────┘
             ▼
┌─────────────────────────────────────────────┐
│ Step 4: Cross-Encoder 重排序          🔑 亮点 │
│ BGE-Reranker-v2-m3 · top-20 → top-k         │
│ 模型不可用 → 自动降级返回原始排序              │
└────────────┬────────────────────────────────┘
             ▼
┌─────────────────────────────────────────────┐
│ Step 5: 上下文构建                           │
│ "## 文档名\n内容片段" · 去重 · Markdown 格式化 │
│ + 🌐 网络搜索结果（如启用）                    │
└────────────┬────────────────────────────────┘
             ▼
┌─────────────────────────────────────────────┐
│ Step 6: LLM 流式生成 → SSE 逐字推送          │
│ 系统提示词 + 上下文 + 历史(最近20条) + 用户问题  │
└────────────┬────────────────────────────────┘
             ▼
┌─────────────────────────────────────────────┐
│ Step 7: 回复持久化（含引用来源 + Token 数）    │
└─────────────────────────────────────────────┘
```

**底部强调**：整个流水线**各环节均可独立配置开关**（通过环境变量或前端设置）
`HYBRID_SEARCH_ENABLED` `RERANKER_ENABLED` `QUERY_REWRITE_ENABLED` `WEB_SEARCH_BACKEND`

### 第 7 页：关键技术实现 —— 降级设计 + 查询重写

**标题**：优雅降级优于完美运行 —— 每个组件都有 Plan B

**上半部分：三层降级策略**：

| 降级层级 | 触发条件 | 降级行为 | 用户体验影响 |
|:---:|:---|:---|:---:|
| L1 | BM25 索引未就绪 | 混合检索 → 纯语义检索 | 精度略降，核心功能不受影响 |
| L2 | Cross-Encoder 加载失败 | 重排序 → 返回 RRF 原始排序 | 精度降低但可用 |
| L3 | 查询重写 LLM 调用失败 | 回退 → 使用原始用户消息 | 多轮对话指代可能不准确 |

**下半部分：查询重写效果对比**：

```
❌ 无查询重写:
  用户第1轮: "请介绍柏拉图的洞穴喻言"
  AI: "洞穴喻言出自《理想国》第七卷..."
  用户第2轮: "它的核心论点是什么？"
  检索系统: 搜索 "它的核心论点" → 无相关结果 ❌

✅ 启用查询重写:
  用户第2轮: "它的核心论点是什么？"
  查询重写: → "柏拉图的洞穴喻言的核心论点是什么？"
  检索系统: 搜索完整的独立查询 → 精准命中 ✅
```

**底部金句**：
> "在 AI 应用中，每个外部依赖都可能不可靠。好的系统不是没有故障，而是每个组件都有降级路径。"

### 第 8 页：SSE 流式传输 + 对话管理增强

**标题**：不止于"一问一答"——SSE 流式输出 + 五大对话管理增强功能

**上半部分：SSE 流式事件协议**：

```
Client (React)                      Server (FastAPI)
  │  POST /api/chat/stream ──────────►  启动 RAG 流水线
  │                                    (查询重写→检索→融合→重排序)
  │  ◄── data: {"type":"chunk","content":"柏"}  逐字推送
  │  ◄── data: {"type":"chunk","content":"拉"}  text/event-stream
  │  ◄── data: {"type":"chunk","content":"图"}
  │  ◄── ... (SSE 连续推送)
  │  ◄── data: {"type":"sources","sources":[...]}  引用来源列表
  │  ◄── data: {"type":"done","conversation_id":"..."} 流结束

  异常路径:
  │  ◄── data: {"type":"error","message":"LLM 服务超时"}  友好错误
```

**下半部分：五大对话管理功能**：

| 功能 | 操作入口 | 技术要点 |
|:---:|:---|:---|

### 第 9 页：系统提示词工程

**标题**：提示词不是"指令"，而是"人格"——苏格拉底式 AI 导师的四层提示词框架

**框图层级结构**：

```
┌──────────────────────────────────────────┐
│  🎭 身份层                                │
│  "你是「苏格拉底」，一位生活在数字时代的     │
│   古希腊哲人导师。你的使命不是灌输答案，     │
│   而是引导学习者走向「理型」的真知。"        │
├──────────────────────────────────────────┤
│  📏 规则层（必须严格遵守）                  │
│  1. 一次只问一个问题                       │
│  2. 每次回复 3-5 句话 + 1 个问题           │
│  3. 不要内联引用（由系统侧面板展示）        │
│  4. 渐进引导，根据用户回答决定下一步        │
├──────────────────────────────────────────┤
│  🎯 方法论层（教学之道）                   │
│  • 苏格拉底式追问：问题→回答→追问           │
│  • 从具体升到抽象：实例→底层原理            │
│  • 跨域联结：知识与其它领域的同构关系        │
│  • 知晓无知：不确定时坦然承认               │
├──────────────────────────────────────────┤
│  📚 知识融合层                             │
│  • 知识库优先（用户提供的权威资料）          │
│  • 训练知识补充（过时时标注 ⚠️ 修正）       │
│  • 网络搜索规则动态注入（当启用 web search） │
│  • 语言自适应（与提问者使用相同语言）        │
└──────────────────────────────────────────┘
```

**设计原则**：
- **行为约束优先**：明确"不能做什么"比"应该做什么"更有效
- **留白设计**：不给过于详细的回复模板，保留 AI 灵活性
- **可观测性**：规则可验证（如"一次只问一个问题"）
- **动态注入**：网络搜索规则按需追加，不污染基础提示词

│  🔄 AI Agent 协作的 Bug 修复闭环             │
│                                             │

**右侧：两种质量保障策略**

| 策略 | 操作方式 | 关键 Prompt |
|:---|:---|:---|
| 🔍 代码审查 | 新对话中让 Agent 以 Reviewer 视角检查 diff | "请审查当前分支的改动，重点关注安全性、错误处理、代码重复" |

### 第 16 页：深度反思 —— AI Agent 开发的三个核心挑战

**标题**：用工程确定性包裹模型不确定性 —— AI 应用开发的三重挑战

**三个核心挑战**：

```
挑战一：模型输出的不确定性
┌────────────────────────────────────────────────────┐
│ LLM 的输出具有随机性，查询重写可能偏离原意            │
│ AI 回复可能不符合苏格拉底风格                        │
│                                                     │
│ 应对策略：                                          │
│ • 所有依赖 LLM 的环节 = 非关键路径 = 自动回退         │
│ • 查询重写失败 → 使用原始消息（不阻塞对话）            │
│ • 风格偏差 → 系统提示词的规则层约束 + 持续迭代         │
│ • 将 AI 的不确定性控制在可接受的有限范围内             │
└────────────────────────────────────────────────────┘

挑战二：外部服务的不可靠性
┌────────────────────────────────────────────────────┐
│ LLM API 超时、Embedding API 限流、搜索服务不可用      │
│ 任何网络波动都可能级联影响用户体验                    │
│                                                     │
│ 应对策略：                                          │
│ • 每个外部依赖 = 超时保护 + 重试机制 + 降级路径       │
│ • Embedding: 指数退避 (0.5s→1s→2s, 最多 3 次)       │
│ • LLM: 120s 超时 + ExternalServiceError → 502        │
│ • 搜索: 多后端 (DDGS→Tavily→Brave→Serper) + 免费兜底  │
└────────────────────────────────────────────────────┘

挑战三：AI Agent 协作中的信任与验证
┌────────────────────────────────────────────────────┐
│ Agent 生成的代码可能包含逻辑缺陷或安全隐患             │
│ Agent 可能产生"幻觉代码"——看起来正确但不 work          │
│                                                     │
│ 应对策略：                                          │
│ • 每个 Agent 输出的代码必须经过实际运行验证            │
│ • 关键功能使用独立的"代码审查"对话进行二次检查         │
│ • 保持 Git 提交粒度小（每次提交可独立回滚）            │
│ • 建立与 Agent 的 Bug 修复闭环（反馈→分析→修复→验证）  │
└────────────────────────────────────────────────────┘
```

**核心论点**：
> 构建 AI Agent 应用的本质挑战不是模型的"智能"不足，而是**系统工程的复杂性**。
> 好的 AI 应用 = 强大的基础模型 × 健壮的工程架构 × 高效的 Agent 协作策略。
